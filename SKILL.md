# Deep Investigate - 深度调查指南

> **兼听则明，偏信则暗** — 多角度、多来源、交叉验证

## 核心原则

**事实是骨骼，分析是灵魂。**

- 不是新闻播报器——要给出独到判断，不人云亦云
- 不是资料库——要拎出关键，忽略噪音
- 分析基于事实 + 理论，不基于情绪或猜测
- 精准，不在长——3句打中要害 > 10句隔靴搔痒

---

## 统一使用 subagent 后台调查

**所有深度调查任务一律使用 subagent**，不阻塞主会话。

原因：主会话被阻塞会导致超时失败，用户体验差；subagent 后台运行不受 timeout 影响，且可以分阶段汇报进度。

### 工作流程

```
用户：深度调查 XXX
主会话：好，我派专项小组去做（spawn subagent）
        ↓
主会话恢复正常，可继续和用户聊天
        ↓
subagent 在后台运行，分阶段发报告到主会话
        ↓
主会话收到 subagent 推送的中间结果，转发给用户
        ↓
用户可以随时说"够了"或"往 XX 方向继续"
        ↓
subagent 完成后推送最终报告
```

### 启动 subagent 的标准话术

```
收到！我派专项小组去做深度调查。
主会话恢复正常，你可以继续和我聊天，结果出来我会通知你。
```

### 启动命令

```python
sessions_spawn(
    task="""【深度调查任务】

调查主题：[用户的具体问题]

调查目标：
1. [具体要回答的问题]
2. [第二个要回答的问题]
3. [如果有第三个]

信息源策略（根据主题调整）：
- 搜索引擎：Bing + 百度
- 权威媒体：[列出具体媒体]
- 预测市场：Polymarket / Manifold（如是实时事件）
- 金融市场：如涉及地缘，查里亚尔/黄金/原油（如相关）
- 社交平台：小红书 / 知乎 / Reddit（如是消费类产品或公众事件）

调查要求：
- 分阶段汇报：每完成一个信息源组合，汇报一次进展
- 交叉验证：关键结论必须有 2+ 不同类型来源确认
- 保持简洁，精准不求长

调查报告格式（严格遵循这个模板）：

好，我已经有足够数据给出分析框架了。
🔍 【深度调查报告：XXX】
**调查时间：YYYY年MM月 | 来源：XXX + XXX + XXX**

---

### 一、[第一个核心发现]

| 数据/指标 | 数值 | 说明 |
|-----------|------|------|

（正文分析——事实→含义→逻辑，简洁精准）

**多渠道交叉验证：**
- 渠道A：[数据/结论] ✅
- 渠道B：[数据/结论] ✅
- 渠道C：[数据/结论] ⚠️（存疑原因）

### 二、[第二个核心发现]

（同上格式）

### 三、[第三个...]

（视情况，可选）

---

### 四、综合判断

**当前形势：[一句话定性]**
- 短期：[方向+逻辑]
- 长期：[方向+逻辑]

**给你的决策参考（如果适用）：**
- 如果你想XXX：[建议]
- 如果你想XXX：[建议]
- 绝对不要：[警示]

**关键跟踪指标：**
1. [指标1]
2. [指标2]

---

**总结：[最终结论，一句话]**

完成调查后，subagent 必须先说：
"我已经收集了足够的信息，深度调查报告如下："
然后再按以上格式输出完整报告。
最后发 sessions_send 到 sessionKey: main，标记 [调查完成]。
""",
    runtime="subagent",
    mode="session",        # 保持会话，可以分阶段汇报
    cleanup="keep",        # 完成后保留，方便查历史
)
```

### subagent 的标准行为

1. **收到任务后**：先给一个"已收到任务，开始调查"的确认
2. **每个阶段完成后**：发 `sessions_send` 到主会话，汇报进展（简短）
3. **有疑问时**：发 `sessions_send` 问主会话，等方向确认
4. **完成后**：
   - 先说："我已经收集了足够的信息，深度调查报告如下："
   - 再按标准格式输出完整报告
   - 最后发 `sessions_send` 到主会话，标记 `[调查完成]`
5. **失败时**：发 `sessions_send` 说明遇到了什么问题

### 主会话收到 subagent 结果后的标准行为

```
收到小组汇报：[subagent 发来的内容摘要]
→ 转发给用户
→ 用户可以回复方向调整或"够了够了"
→ 如果用户没回复且 subagent 已标记完成，可以结束 subagent 会话
```

## ⚠️ 链接反查（必须掌握的流程）

### 什么时候需要反查？

**抓取搜索结果后，所有链接都会变成代号，必须反查才能拿到真实 URL。**

典型场景：
- 抓取微博/小红书/知乎搜索结果后，想访问具体帖子
- 批量抓取后处理结果文件，想知道每个链接指向什么
- 想抓取特定帖子详情，但只有搜索结果页

### 反查链接

```bash
# 反查单个链接
python get_link.py @abcd-1

# 反查多个（推荐，一次性反查多个）
python get_link.py @abcd-1 @abcd-2 @abcd-3
```

### 完整使用流程

```bash
# 1. 抓取搜索结果（假设搜索微博）
python fetch.py "https://s.weibo.com/weibo?q=关键词" -s session.json -w 5

# 2. 输出中链接都是代号，如 @abc-12
#    复制需要反查的代号

# 3. 反查获取真实 URL
python get_link.py @abc-12 @abc-15 @abc-20

# 4. 输出：@abc-12 -> https://weibo.com/1234567890
#    用真实 URL 访问帖子详情
python fetch.py "https://weibo.com/1234567890" -s session.json -w 5
```

**注意**：代号 `@{namespace}-{number}` 中的 namespace 是每次抓取随机生成的，旧的代号在新的抓取中可能失效（但数据库中已保存的映射不受影响）。

### 数据库清理

```bash
# 清空全部（谨慎使用）
python get_link.py --clear

# 删除特定链接
python get_link.py --clear @abcd-1

# 删除某命名空间下所有链接
python get_link.py --clear abcd
```

**清理要求：**

- `links.db` 会自动清理 7 天前的旧数据（5% 概率触发）
- 抓取大量页面后，可手动执行 `--clear` 释放空间
- 清理后无法反查之前的链接代号

## 调查流程

```
1. 问题定义 → 2. 信息源规划 → 3. 批量抓取
                                              ↓
6. 总结报告 ← 5. 交叉验证 ← 4. 内容分析
                    ↑
            4.5 深度分析（最关键步骤）
```

**注意：**
- 事实核查型任务，必须包含预测市场信号（Polymarket/Manifold）
- 实时事件（正在发生的事），必须查金融市场信号（货币/黄金/股市）
- 不同类型的问题，用不同的信息源组合
