# 调查方法论

## 一、问题定义

在开始调查前，先明确：

| 要素 | 说明 | 示例 |
|------|------|------|
| **核心问题** | 你到底想知道什么？ | "X 公司是否真的裁员 30%？" |
| **时间范围** | 关注什么时间段的信息？ | "最近 1 个月" |
| **地理范围** | 关注哪些地区？ | "中国区" |
| **信息类型** | 需要什么类型的证据？ | "官方公告、员工爆料、财经数据" |

## 二、信息源选择策略

| 信息类型 | 优先选择 | 避免 |
|---------|---------|------|
| **事实数据** | 官方公告、财报、权威媒体 | 匿名论坛、小道消息 |
| **市场反应** | 股价、交易量、预测市场 | 单一自媒体解读 |
| **民间声音** | 多个独立信源交叉验证 | 单个"内部人士" |
| **历史记录** | Wayback Machine、存档站 | "我记得"式口述 |

## 三、调查流程

```
1. 问题定义 → 2. 信息源规划 → 3. 批量抓取
                                              ↓
6. 总结报告 ← 5. 交叉验证 ← 4. 内容分析
```

## 四、实战检查清单

每次调查前，确认你覆盖了：

- [ ] 问题定义清晰，知道要找什么
- [ ] 规划了至少 3 类不同信息源
- [ ] 包含官方渠道 + 第三方独立来源
- [ ] 准备了 URL 列表，使用批量抓取
- [ ] 有交叉验证的计划
- [ ] 考虑了"历史快照"等存档手段
- [ ] 识别了可能的利益冲突方
- [ ] 记录了矛盾点并尝试解释
- [ ] 结论有证据支撑，不做过度推断

## 五、"产品全景图"类调查专项清单

当调查目标是**找全所有同类产品/服务**（如"Coding Plan 有哪些平台"）时，必须执行以下步骤：

### 5.1 搜索引擎多轮搜索

**第一轮：找已知玩家**
```
搜索词：{产品名} 订阅 套餐
示例：Coding Plan 订阅 套餐
```

**第二轮：找"还有谁"**
```
搜索词：{产品名} 还有哪些平台
搜索词：{竞品A} {竞品B} {产品名} 对比
搜索词：国内 {产品名} 平台 有哪些
```

**第三轮：反向验证**
```
搜索词：{产品名} 评测 对比 哪个好
搜索词：用了 {产品名} 怎么样
```

### 5.2 权威来源覆盖检查

每个平台至少需要通过 **2 个独立来源** 确认其存在，不可只靠一个搜索结果。

| 平台类型 | 必须查证 |
|---------|---------|
| 云厂商（阿里/腾讯/字节/百度） | 官方产品页 + 第三方报道 |
| 创业公司（月之暗面/智谱） | 官网 + 科技媒体 |
| 运营商（联通等） | 官方公告 + 社区讨论 |

### 5.3 常见遗漏原因与应对

| 常见遗漏 | 应对方法 |
|---------|---------|
| 搜索词不够宽 | 用"还有哪些平台"类搜索补全 |
| 官网改版/404 | 尝试 `site:zhihu.com` 或科技媒体 |
| 页面需要登录 | 换第三方报道来源 |
| 反爬虫拦截 | 换 Bing 国际版，或用 `textise dot iitty` 参数 |
| 内容被删除 | 查 Wayback Machine 历史快照 |

### 5.4 交叉验证要求

在最终报告前，必须自问：
- [ ] **找到的平台数量是否达到预期？**（如：预期5-8家，实际找到几家？）
- [ ] **每个平台是否都有2+独立来源？**
- [ ] **有没有"这家也有这个产品"的线索被忽略？**（如：文章提到"Kimi 也推出了 Code Plan"）
- [ ] **有没有平台被遗漏是因为"名称不熟悉"？**（如：ArkClaw 就是字节跳动的火山引擎）
